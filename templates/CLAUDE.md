# CLAUDE.md

このファイルはコーディングエージェント(Claude Code / Codex 等)向けの指示書である。
`AGENTS.md` はこのファイルへのsymlinkであり、内容は同一。

このプロジェクトでは、作業を通じて得たドメイン知識と設計判断を
リポジトリ内に蓄積する。その更新はエージェントであるあなたの責務である。
以下のルールに従うこと。

---

## 着手前に必ず読む

新しいタスクに着手する前に、以下を読んでから作業を始めること。

1. `knowledge.md` — このプロジェクト固有のドメイン知識・ハマりどころ
2. `docs/adr/` 配下の関連するADR — 過去の設計意思決定とその理由
3. 横断的な進め方・組織共通のノウハウは **Notion の FDE Playbook** を参照

これらを踏まえずに設計や実装を進めてはならない。過去の決定を覆す可能性がある場合は、
該当するADRを必ず確認すること。

---

## knowledge.md の更新ルール

`knowledge.md` は「常に最新であってほしい、このプロジェクト固有の生きた知識」を置く場所。

### 何を書くか

- このクライアント/ドメインの仕様の癖、業務ルール、用語の定義
- API・外部システムの制限、想定外の挙動、ハマりどころ
- デプロイ・運用・テスト実行の注意点
- 「次に作業する人(人間でもエージェントでも)が知らないと詰まること」

### 何を書かないか

- コードを読めば分かること(構造・関数の説明など)。重複は避ける。
- 一度きりの作業ログ、このタスク限りの一時的な事情
- 設計の意思決定とその理由 → これは knowledge.md ではなく ADR に書く(後述)

### 更新の仕方

- **常に最新化する。** 古くなった記述は履歴を残さず上書きして正しい状態にする
  (履歴はGitが持つので、ファイル内に古い情報を残さない)。
- タスク完了時に、今回得た知見で `knowledge.md` を更新する差分を提案する。
- 提案した差分は人間のレビューを経てコミットする。勝手にコミットしない。
- 既存の記述と重複・矛盾する内容を追記しない。該当箇所を編集して統合する。

## ADR(設計意思決定記録)の更新ルール

`docs/adr/` には「なぜその設計を選んだか」を決定ごとに1ファイル・連番で残す。
フォーマットは `docs/adr/0000-template.md` に従う(MADR形式)。

### いつADRを起こすか

判断基準は **「将来この決定を覆すとき、過去の理由を残す必要があるか?」**。

- Yes → ADRを起こす。例: 採用する技術・ライブラリ・アーキテクチャの選定、
  データモデルの方針、認証方式、却下した代替案を残すべき判断。
- No → ADRではなく `knowledge.md` に書く(運用知識・最新化される情報)。

設計上の分岐点で意思決定を行ったら、実装に入る前か直後にADRを起こすこと。

### 更新の仕方

- 新しい決定ごとに `0001-xxx.md`, `0002-xxx.md` と**連番で新規ファイル**を作る。
  番号は既存の最大値+1。ファイル名は決定内容を表すkebab-caseにする。
- **既存のADRは書き換えない。** 一度 Accepted になった決定の本文は編集しない。
- 過去の決定を覆す場合:
  1. 新しいADRを起こし、そこに新しい決定と理由を書く。
  2. 古いADRの Status を `Superseded by ADR-XXXX` に変更する
     (Statusの変更だけは既存ファイルへの編集として許容する)。
- ADRの新規作成・Status変更も、人間のレビューを経てコミットする。
